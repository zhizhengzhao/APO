# APO 三类任务最终实验报告：Code / Math / Reasoning

生成日期：2026-06-15  
报告范围：本报告只覆盖三个最终保留类别：`code`、`math`、`reasoning`。所有结论均基于最终训练产物重新解析得到；旧的跨实验统计文件没有作为本报告的数据源。

## 0. 一页结论

三个类别均已完整完成 18 epoch 等价训练进度：每类 `history.json` 均包含 450 条 step 记录（`step=0...449`），且最终 checkpoint 均已落盘为 `head_grpo_step450`。最后 20 步没有 API error，训练闭环健康。

核心结论如下：

| 类别 | 完成状态 | 最终近端正确率 | 最终平均 active agent | 最终平均 LLM calls | 学到的主架构 A |
|---|---:|---:|---:|---:|---|
| code | 450/450 + ckpt450 | 77.3% | 2.26 | 4.65 | 2-agent 为主：`Planner[flash] + Solver[max]`，2 条边，基本是双向协作 |
| math | 450/450 + ckpt450 | 72.1% | 2.09 | 5.82 | 1-2 个 `Solver` 为主，`plus` 偏好最高，通信边大多为 0 |
| reasoning | 450/450 + ckpt450 | 73.3% | 2.91 | 4.55 | 2-3 agent 为主：`Solver[plus]` 核心，常加 `Verifier/Refiner`，通信更密 |

最重要的科学结论是：APO 没有学出“一套通吃”的固定多智能体结构，而是明显按任务类型学习了不同的架构策略。代码类偏“便宜规划 + 强模型求解 + 互看”；数学类偏“低通信、少 agent、plus 求解器”；推理类偏“plus 求解器集群 + 检查/修订角色 + 中高通信”。

## 1. 实验目标

本项目的目标是学习一个任务条件化的 architecture policy。给定一个任务文本，policy head 不直接生成答案，而是采样一个多智能体架构 `A`：

- 选择激活多少个 agent；
- 为每个 agent 选择 role，例如 `Planner`、`Solver`、`Verifier`、`Refiner`；
- 为每个 agent 选择 worker model，即 Qwen 三档模型之一；
- 选择 agent 的执行顺序；
- 选择有向通信边，即哪些 agent 的输出对后续/其他 agent 可见。

训练时固定底层 worker pool，只学习 architecture head。这样可以回答一个核心问题：在不同任务类型上，系统到底偏好什么样的多智能体组织形式。

## 2. 数据集与难度控制

三个最终类别均使用冻结的 500 题 corpus。类别数据通过 `CategoryBench` 插件加载，算法和数据源分离：新增类别只需要新增/替换 `data/categories/<cat>.jsonl`，训练算法本身不需要改。

| 类别 | 题数 | train/test | 探针 pass_rate 均值 | 有方差题数量 | sat0 / sat1 | 主要来源 |
|---|---:|---:|---:|---:|---:|---|
| code | 500 | 399/101 | 0.627 | 275 | 75 / 150 | livecodebench 405；mbpp 47；humaneval 48 |
| math | 500 | 397/103 | 0.618 | 253 | 86 / 161 | omni_math 237；olympiad 125；aime 86；phybench 52 |
| reasoning | 500 | 398/102 | 0.536 | 302 | 82 / 116 | musr 142；reclor 52；arc 35；BBH families 271 |

难度探针的设计重点是让 step-0 附近存在足够的架构区分度：

- 使用真实 Qwen 三档 worker pool 跑候选题；
- 架构从共享 SFT prior 中采样；
- 每个 active slot 的模型以 1/3 均匀采样；
- 用和训练相同的 reward/grade 路径计算 pass rate；
- 最终每类固定 500 题，并按 pass rate 分层形成 train/test。

这里的 `sat0` 表示探针下全错题，`sat1` 表示探针下全对题；`0 < pass_rate < 1` 的题是最有直接 GRPO 区分信号的题。

## 3. 模型池与运行配置

三个类别使用同一套 Qwen worker pool：

| 简称 | 实际模型 | 作用直觉 |
|---|---|---|
| flash | `qwen3.6-flash` | 便宜、快，适合规划、粗筛、简单 reasoning |
| plus | `qwen3.6-plus` | 中间能力/成本，数学与推理中成为主力 |
| max | `qwen3.7-max` | 最强、最贵，代码求解中被明显偏好 |

所有 worker 和 judge 都使用 non-thinking 路径。Qwen worker 默认 `thinking=False`，请求里显式传 `extra_body={"enable_thinking": False}`。这样实验学习的是多智能体结构和模型分配，而不是让某个模型通过隐藏 thinking 获得额外优势。

训练脚本核心参数：

| 参数 | 值 |
|---|---:|
| 初始 head | shared SFT prior：`sft_shared_v2/head_final` |
| worker pool | `qwen3.6-flash,qwen3.6-plus,qwen3.7-max` |
| judge | `qwen3.7-max`，thinking off |
| batch size | 16 |
| GRPO group size | 4 |
| epochs | 18（最终三类均完成 450 steps） |
| learning rate | `3e-5` |
| LoRA rank | 32 |
| `cost_bonus_scale` | 0.5 |
| max concurrent runs | 64 / job |
| save interval | every 25 steps |
| code max_new_tokens | 8192 |
| math python timeout | 120s |

## 4. Reward 与 Advantage 设计

最终 advantage 采用两层逻辑。第一层先给出原始正确性优势：

- 错误样本：`raw = -1`；
- 正确样本：`raw = 1 + cost_bonus`；
- `cost_bonus` 只在正确样本内部根据 `n_calls` 做 min-max，越省调用 bonus 越大；
- 本轮最终 `cost_bonus_scale = 0.5`，所以正确样本范围是 `[1, 1.5]`。

第二层按同一个 task 的 group 组成决定是否产生梯度：

| group 情况 | advantage 处理 | 目的 |
|---|---|---|
| 全对 | `adv = 0` | 架构不改变正确性，不让成本梯度压过准确率信号 |
| 全错 | `adv = -0.1` | 给一个很小的负推力，鼓励离开无效架构 |
| 有对有错 | `adv = raw / std(raw_valid)` | 保留正确性主信号，同时让正确样本内部的低调用数成为 tie-breaker |

这点很关键：成本不是独立优化目标，而是“在能做对的架构之间”作为轻量偏好。这样避免了全对 group 中仅因 `n_calls` 差异产生巨大 cost gradient，导致准确率被成本优化反向侵蚀。

同时，工程无效样本通过 `eng_valid` mask 排除：API error、关键截断、judge-side infra error 等不会作为“模型答错”污染 policy gradient。

## 5. 通信图语义

通信图是有向边矩阵 `edges[src, dst]`。执行时，`dst` agent 能看到 `src` 的消息，当且仅当存在直接边 `src -> dst`。可见性不是传递闭包。

也就是说，如果存在 `A -> B` 和 `B -> C`：

- `B` 可以直接看到 `A`；
- `C` 可以直接看到 `B`；
- `C` 不会因为 `A -> B -> C` 自动看到 `A`，除非也存在直接边 `A -> C`；
- 信息当然可以被 `B` 在自己的文本输出中复述给 `C`，但那是内容传播，不是结构上的直接可见性。

本报告对通信部分使用两类信息：

- `edges_count`：可靠，表示该架构有多少条有向边；
- 典型通信图示意：用于解释最常见架构 family 的功能含义。

注意：当前最终 `details.jsonl` 可靠记录了 `edges_count`，但没有完整保存每个 sample 的 edge matrix。因此，除了 2-agent 且 `edges_count=0/1/2` 这种可以严格解释的情况，本报告不声称可以精确复原每一条具体边的方向。

![训练曲线](images/fig1_training_curves.png)

从训练曲线看，三类共同呈现两个趋势：

- `n_calls` 明显下降，说明 policy 学到了更省调用的架构；
- `n_active` 明显下降，说明 policy 从较大的探索性团队收缩到更小、更稳定的任务特化团队。

准确率曲线的解释要更谨慎：这不是固定 validation set 上的单调曲线，而是 on-policy training batch 的 correct rate，因此会有抽样噪声。即便如此，最终近端正确率维持在较高区间，并且没有出现训练后期崩塌。

## 7. 学到的模型选择

![模型选择](images/fig2_model_share.png)

三类对模型的偏好不同：

| 类别 | flash | plus | max | 解释 |
|---|---:|---:|---:|---|

## 9. Active Agent 数与通信密度

![active 与 edges 分布](images/fig4_active_edges.png)

| 类别 | n_active 主峰 | edges_count 主峰 | 解释 |
|---|---|---|---|
| code | 2 agents（69.8%） | 2 edges（69.7%） | 两个 agent 双向互看最常见 |
| math | 2 agents（61.4%），其次 1 agent | 0 edge（74.8%） | 独立求解/投票，比互相通信更稳定或更省 |
| reasoning | 3 agents（45.3%）和 2 agents（31.0%） | 1、4、5 edges 较常见 | 比 math 更需要多角色交互与检查 |

通信密度是区分三类学到的 A 的核心证据之一。Code 的 2-agent + 2-edge 很明确：两人双向可见。Math 的 0-edge 主峰也很明确：agent 间不共享中间文本，Synth 聚合结果。Reasoning 的边数更高，说明它更依赖多 agent 的中间结果交流，但由于最终日志没有完整 edge matrix，报告只讨论通信密度，不臆造每条边方向。

## 10. 典型学到的 A

![典型架构 family](images/fig6_typical_architecture_families.png)

### 10.1 Code：Planner[flash] + Solver[max]

最终 30 步最常见架构 family：

| rank | family | n_active | edges_count | count |
|---:|---|---:|---:|---:|
| 1 | `Planner[flash] + Solver[max]` | 2 | 2 | 400 |
| 2 | `Solver[flash] + Solver[max]` | 2 | 2 | 213 |
| 3 | `Planner[max] + Solver[max]` | 2 | 2 | 117 |
| 4 | `Solver[max] + Solver[max]` | 2 | 2 | 93 |
| 5 | `Planner[plus] + Solver[max]` | 2 | 2 | 85 |

Code 学到的是清晰的异质协作结构：

- `Planner[flash]`：低成本模型先拆任务、提出实现路线、指出边界条件；
- `Solver[max]`：强模型负责核心代码生成、复杂逻辑和修复；
- `edges_count=2` 且 `n_active=2`，这在 2-agent 情况下等价于双向通信。

这符合代码任务的特点：任务通常有工程结构和边界条件，规划与实现分离有帮助；同时代码生成质量对模型能力敏感，所以 `max` 被放在 Solver 位置。

### 10.2 Math：Solver[plus] 为中心，低通信

最终 30 步最常见架构 family：

| rank | family | n_active | edges_count | count |
|---:|---|---:|---:|---:|
| 1 | `Solver[plus] + Solver[plus]` | 2 | 0 | 80 |
| 2 | `Solver[plus]` | 1 | 0 | 73 |
| 3 | `Solver[max] + Solver[plus]` | 2 | 0 | 72 |
| 4 | `Solver[flash] + Solver[plus]` | 2 | 0 | 65 |
| 5 | `Solver[max]` | 1 | 0 | 49 |

Math 学到的是低通信、低 agent 数的结构：

- `edges_count=0` 占 74.8%；
- `n_active=2` 占 61.4%，`n_active=1` 也有 16.5%；
- `plus` 是模型主力，占 49.7%。

这说明当前数学任务中，多 agent 之间反复通信并没有稳定带来更高收益；更好的策略是少量独立 solver 产生候选解，由 Synth 进行最终整合。数学类的准确率变化不如 `n_calls/n_active` 那样单调，但成本和结构收缩非常明显。

### 10.3 Reasoning：Solver[plus] + Verifier/Refiner

最终 30 步最常见架构 family：

| rank | family | n_active | edges_count | count |
|---:|---|---:|---:|---:|
| 1 | `Solver[plus] + Solver[plus]` | 2 | 1 | 405 |
| 2 | `Solver[flash] + Solver[plus]` | 2 | 1 | 60 |
| 3 | `Solver[plus] + Solver[plus] + Verifier[flash]` | 3 | 4 | 59 |
| 4 | `Solver[plus] + Solver[plus] + Verifier[plus]` | 3 | 4 | 52 |
| 5 | `Solver[plus] + Solver[plus] + Verifier[max]` | 3 | 4 | 50 |

Reasoning 的结构介于 code 和 math 之间，但更接近“多 agent 检查/修订”：

- `plus` 占 72.5%，是非常明确的主力模型；
- `Solver` 占 68.4%；
- `Verifier` + `Refiner` 合计占 27.1%；
- `n_active=3` 和 `n_active=2` 是主峰；
- edges_count 的分布比 math 密得多，说明 reasoning 更需要中间结论互相影响。

直观解释是：推理题往往不是纯计算，也不是纯代码实现，而是需要消除歧义、验证选项、修订初始判断。因此 policy 学出了 `Solver` 主导、`Verifier/Refiner` 辅助的模式。

## 11. 收敛性判断

从最终三类曲线和统计看，可以认为三类都已经进入可分析的稳定阶段，理由如下：

1. checkpoint 完整：三类都有 `head_grpo_step450`。
2. API 稳定：三类最后 20 步 API error 均为 0。
3. 结构稳定：三类的 `n_active_mean` 都从初期较高值下降并稳定在任务特定区间。
4. 成本稳定下降：`n_calls_mean` 在三类中均明显下降，说明 cost tie-breaker 没有失效。
5. 模型选择明显非均匀：三类从 SFT 的近似均匀模型 prior 中分化出来，code 偏 `max`，math/reasoning 偏 `plus`。
6. role 结构有任务差异：code 的 `Planner` 显著高于 math/reasoning；reasoning 的 `Verifier/Refiner` 显著高于 code；math 的通信最低。

需要强调的是，准确率不是严格 validation 曲线，而是 on-policy batch correct rate。因此不能把单步上下波动解释成泛化性能涨跌；更可靠的是最后窗口均值、结构分布和成本趋势。

## 12. 工程可靠性与断点续训

训练保存机制包含：

- 每 25 steps 保存 `head_grpo_step*`；
- 保存 `resume.pt`，包含模型参数、optimizer 状态、last completed step、RNG 状态和累计时间；
- 最终三类均检查到 `head_grpo_step450`；
- 训练中断后可以从最近完整 checkpoint 继续，最终三类均已经完成到统一长度。

这里的“完美”含义是工程意义上的完整续训：不会丢模型/optimizer 状态，不会从头重跑，不会把坏的半行 JSON 当作有效训练结果。严格 bit-identical 的物理确定性不保证，因为远端 API 响应、并发完成顺序和 CUDA kernel 本身都可能有非确定性。但对本项目的目标（拿到一个完整、可用、无污染的最终 ckpt）来说，三类最终结果是完整的。

## 14. 最终结论

三个最终类别的 APO 训练是有效且完成的。最核心的结果不是单纯正确率，而是 policy head 学出了任务条件化的架构偏好：

- Code：`Planner[flash] + Solver[max]`，2-agent 双向通信，体现“便宜规划 + 强模型实现”。
- Math：`Solver[plus]` 为中心，少 agent、低通信，体现“独立求解 + Synth 聚合”。
- Reasoning：`Solver[plus]` 主导，常配 `Verifier/Refiner`，通信更密，体现“求解 + 检查/修订”。

这三个结果共同支持 APO 的主要论点：多智能体系统的最优架构不是固定模板，而应当随任务类别、模型能力/成本和通信需求自适应选择。
